MODULE 01

产品概述

飞书 AI Agent 是一个面向自动驾驶研发场景的智能工作台。它以飞书为企业协作入口， Claude API 为推理引擎，深度集成 ESS（车辆数据平台）、Mviz（可视化分析）、FMP（车辆管理）等 领域系统，打通「车辆故障诊断 → Bag 数据挖掘 → 知识库 RAG 检索 → 项目计划生成 → 国标条款检索」的完整研发链路。 同时覆盖会议邀约、知识库 RAG、群聊摘要等通用办公场景。系统上线 10 个月，累计处理消息 10,000+ 条， 服务团队 50+ 人。

核心业务场景

业务场景传统痛点Agent 驱动的闭环
车辆故障诊断 诊断工程师拿到 VIN → 登录 ESS 平台 → 人工筛选 Bag → 逐一比对 23 万条故障码表 → 手工撰写诊断报告。单次耗时 15-30 分钟。 飞书输入「查 VIN xxx 昨天的 enable_signal_cmd」→ Agent 自动定位对应 Bag → 解析故障 Topic → 匹配故障码映射表 → 输出结构化报告（故障名称 / 受影响功能 / 降级模式 / 严重度），耗时 < 5 秒。支持交叉分析——同时段多个故障码自动关联推理根因。
Mviz 数据挖掘 算法工程师定位问题需打开 Mviz Web → 搜索 Stream → 查找 ROS Topic → 下载数据 → 本地分析。多步操作分散在 3 个系统。 飞书对话中直接搜 Stream → 查 Topic 定义 → 下载指定帧数据 → Claude 基于数据给出分析结论。四步操作一条消息完成，中间结果自动缓存复用。
飞书知识库 RAG 检索 团队知识分散在飞书 Wiki、云文档、多维表格、妙记等不同载体中。传统检索需逐一切换应用 → 关键词搜索 → 人工翻阅 → 跨文档关联依赖个人记忆。 Agent 在对话中直接理解查询意图 → 调用 feishu_action(search) 全局搜索 → read_page 获取全文 → Claude 基于原文回答。轻量级 RAG：不做离线 Embedding 索引，而是实时搜索 + 全文注入 Prompt，兼顾准确性和零维护成本。
项目排期生成 项目经理根据 SOP 模板手工排甘特图，需对齐 T0/T+4W/T+10W/T+18W/T+20W 里程碑，涉及底盘联调/控制匹配/相机/Radar/超声/域控/CP主线等 10+ 模块的时间线编排。 飞书输入「生成 XX 项目二级计划，T0=2026-08-01」→ Agent 自动生成 HTML 甘特图 + HD PNG，按标准里程碑编排全部模块任务，一键上传飞书云盘和 Wiki。支持自定义里程碑和模块覆盖。
车辆预约与调度 测试工程师需手动查看 FMP 平台空闲车辆 → 在群内发消息预约 → 等待审批。信息不对称导致车辆利用率低。 「查哪些车空闲」「帮我预约 Vehicle-C xxx 明天 9-12 点」→ Agent 查询 FMP 实时状态 → 自动向 Fleet-Bot 发预约请求 → 审批结果即时反馈。
GB 强标合规检索 工程师遇到合规疑问 → 翻阅 PDF 版 GB 报批稿 → 关键词搜索定位条款 → 人工判断适用性。搜索精度依赖个人经验。 飞书对话中自然语言提问「L2++ 领航功能对 TTC 的要求是什么」→ Agent 检索 GB 强标原文条款 → 基于条款原文作答 → 标注条款编号和出处。不得凭记忆回答（System Prompt 硬约束）。

此外，系统还覆盖会议邀约（search → availability → create → RSVP 追踪全流程）、群聊摘要（六级活跃度分级调度）、非 Owner 权限隔离等通用协作场景。

MODULE 03

核心设计决策

设计原则

🎯

Prompt 即代码

每个 Skill 就是一段注入 LLM 的指令文本。新增能力 = 新增一个 SKILL.md + 触发词列表，不改核心循环。

⚡

分级响应

不是所有消息都需要最强模型。"你好"走 Haiku 秒回，复杂任务走 Sonnet+工具。路由策略节省 60% API 费用。

🛡️

边界优先

19 条格式规则 + 防呆机制不是在设计阶段想出来的，是 10 个月生产运行中逐条追加的。

关键决策

决策点我们的选择理由与权衡
通信协议 WebSocket 长连接 IM 消息要求实时推送。HTTP 轮询模式在 50+ 用户规模下延迟高且浪费带宽。飞书官方 SDK 封装了重连和心跳逻辑。
LLM 引擎 Claude API (Anthropic) 同期 Function Calling 准确率优于 GPT-4，中文理解更自然。Opus/Sonnet/Haiku 三级模型可灵活路由。
工具粒度 聚合型：1 个大工具含 22 个 action 如果飞书 22 种操作各自独立为 Tool，Tool List 膨胀到 40+。聚合后保持 17 个 Tool，Claude 先选工具再选 action，两级决策更精准。
Skill 加载 触发词匹配 → 动态注入 Prompt 始终加载全部 Skill 会让 Prompt 膨胀到 5000+ 行。触发词在应用层匹配（含历史对话上下文），零延迟零成本。
上下文窗口 每用户保留 6 轮对话 太少丢失上下文（用户说"好"无法关联上轮），太多浪费 Token。6 轮覆盖 90% 的多轮交互场景。
单实例保障 PID 文件锁 + SIGTERM → SIGKILL 多实例并发会导致共享 Token 文件损坏。先优雅终止，1 秒后强制杀死。atexit 注册清理函数。

模型路由策略

用户消息 ├─ 含图片? → 跳过 Haiku（不支持视觉）→ Sonnet ├─ "你好"/"谢谢" → simple → Haiku 4.5 (零工具, 精简Prompt, <1s) │ └─ 失败 → 自动降级 Sonnet（用户无感知） ├─ "PPT"/"bag分析"/"需求三抓" → complex → Sonnet 4.6 ├─ "创建"/"发送"/"生成" → write → Sonnet 4.6 └─ 其他查询 → read → Sonnet 4.6 效果: simple 占 40% 消息量，综合节省 ~60% API 成本

Skill 体系

Skill 是本系统的核心能力单元。每个 Skill 由两部分组成：SKILL.md（Prompt 指令，告诉 LLM 何时触发、如何操作） 和源码（Python 实现，执行实际的 API 调用与数据处理）。两者分离使得新增 Skill 无需改动核心 Agent 循环。

Skill 触发机制

设计思路：基础能力始终在 System Prompt 中，专项 Skill 通过触发词匹配动态注入。 匹配在应用层完成（用户消息 + 对话历史），不额外调用 LLM，零延迟零成本。 例如用户说「Project-MXS 现在进展怎么样」→ 命中 CPM 触发词 → 注入 CPM Skill 指令 + 预加载的周例会内容。 新增一个 Skill 只需三步：写 SKILL.md → 定义触发词列表 → 注册到 assistant.py，核心循环无需改动。

典型案例：Bag Fault Query 深度拆解

这是本系统集成深度最深、部署考量最完整的 Skill。单次查询跨越 4 层外部系统， 展示了 Agent 如何在自动驾驶研发场景中实现「一句话替代 15 分钟手动操作」。 更关键的是，它内置了环境自检机制，让任何工程师都能一键验证部署环境是否就绪。

数据管线

用户输入: "查 VIN LSVUM40Z6MN123456 7月15号有没有故障" ① VIN 解析 — 17位 VIN 正则校验，提取厂牌/车型/年款元信息 ↓ ② ESS 事件定位 — query_bag_fault() → ESS API 输入: VIN + time_range(2026-07-15) 输出: 匹配的 Bag 列表（event_id / 时间戳 / 存储路径） 关键: size 参数控制查询范围（默认20），避免海量回扫压垮 ESS ↓ ③ Topic 数据解析 — 定位 Bag → 读取 /mff_md/enable_signal_cmd topic 提取故障码 onset 事件（signal_name + timestamp + value） 依赖: pybagmining + roslz4（LZ4 解压）— 需要在 Linux 环境运行 ↓ ④ 故障码映射 — 查表 error_code_map.json（239,762 条） "0x12345678" → {name:"ESP_FAULT", function:"车身稳定控制", degrade:"ESP功能受限, 限速80km/h", severity:"功能性故障"} ↓ ⑤ 报告生成 — Claude 基于结构化故障列表做因果推理 例如: ESP_FAULT + CAN_TIMEOUT 同时出现在 08:23 → 推断"底盘通信超时触发 ESP 安全降级"

环境自检机制

为了让其他团队也能独立部署这个 Skill（而非依赖 Bot 开发者手动配置）， 内置了 --check 模式：8 项逐层检查，每项附带修复指引。

MODULE 05

工程特色机制

以下 10 个机制是在 10 个月生产运行中逐步迭代出来的，它们让系统从「能跑」变成「可靠」。

1

RSVP 实时追踪 + 拒绝即时通知

创建会议后自动登记到待追踪列表（持久化 JSON），后台每 5 分钟轮询飞书日历 API 检查与会者 RSVP 状态。有人拒绝 → 立刻 DM 通知会议发起人并提示重新安排。事件超过 7 天自动过期清理。

2

Tiered 群聊摘要调度

50+ 个群不能全量日报（太贵太吵）。按群活跃度分六级：HOT（≤1天）→ 日报；ACTIVE（1-7天）→ 周报；WARM（7-30天）→ 双周报；COOL（30-90天）→ 月报；COLD（90-180天）→ 季报；ZOMBIE（>180天）→ 跳过。摘要发布到飞书 Wiki 并 DM 通知。

3

跨租户用户智能处理

飞书日历 API 遇到跨公司用户时会静默跳过（不报错）。系统对比「请求添加」vs「API 实际返回」的用户列表，发现差异后强制 Claude 一次性收集所有外部邮箱（而非逐个询问），用邮件方式重新邀请。

4

SQLite 消息缓存（被踢出群仍可查历史）

所有进入 Bot 的消息实时写入 SQLite，保留 14 天。Bot 被移出群后再重新加入时，摘要功能从本地缓存读取历史，不依赖飞书 API。P2P 对话的 chat_id 映射也持久化到 JSON 文件，跨重启保留。

5

三层 Token 自动刷新 + 启动预热

Tenant Access Token（提前 60s 续期）、User OAuth Token（过期自动 refresh）、feishu-sync Token（每 90 分钟后台预热）。启动时立即执行一次预热再开启 WebSocket，确保首条消息不因 Token 过期而失败。

6

非 Owner 权限隔离

群内其他成员 @Bot 时使用独立的精简 System Prompt（仅告知已转告，不暴露工具能力），零工具调用。无法以 Owner 身份执行任何操作。

7

CPM 周例会预加载

后台每 4 小时搜索最新周例会页面 → 读取前 4000 字摘要 → 缓存。CPM 触发词命中时直接注入 Prompt，零额外延迟即可获取项目最新状态。

8

用户目录启动预热

FeishuClient 初始化时后台线程全量拉取企业通讯录（~3000 人）并缓存。search_users 后续直接在内存做本地模糊匹配，响应从 3-5 秒降至毫秒级。

9

撤回消息安全策略

硬编码优先级：先从对话历史中查找已记录的 message_id → 找不到才告知无法撤回。禁止去群聊历史搜索（可能找错消息撤错）。

10

单实例守护 + CLI 调用串行化

PID 文件锁防止多 Bot 实例并发（共享 Token 被损坏）。feishu-sync-cli 的所有调用经过全局线程锁，防止并发子进程同时读写 Token 文件。

如果重来，会做什么不同

过度设计的地方

  • 初期就做了 RSVP 轮询——前 3 个月根本没人创建会议，白跑了 90 天的空轮询。应该等使用数据验证后再加。
  • Tool Use 循环设了 16 轮上限——实际最多用到 6 轮。过度保守的配置暴露了对 LLM 能力的不信任。
  • 用户目录全量预热——3000 人的企业，首次搜索 3 秒延迟完全可以接受。预热是过早优化。

该做而没做的事

  • 用量仪表盘——没有统计哪个工具被调用最多、哪个模型路由占比最高。靠感觉判断系统健康度。
  • A/B 测试框架——每次改 System Prompt 都是全量上线，改完盯着群聊看反馈。应该先灰度 20% 用户。
  • 成本归因——知道总 API 花了多少钱，但不知道每类任务的单价。无法量化"这个 Skill 值不值得继续维护"。
一句话：前期的坑在 API 边界（日历静默丢弃、Token 竞态、多实例）， 中期的坑在 LLM 行为（幻觉、误解、单字判断）， 后期的坑在规模化边缘 case（合并转发、被移出群、分级调度）。 一个好的 Agent 不是设计出来的，是用出来的——但用完之后要停下来想，哪些是真正的工程问题，哪些只是过早优化。

MODULE 10

后续改进方向

基于 MODULE 08-09 审计出的短板，以下提出可落地的改进方案。按优先级分三档： P0（系统可靠性底线）、P1（工程化成熟度）、P2（规模化支撑）。

P0 · 可靠性底线

改进项投入方案
进程自动守护 低（30 分钟） Windows 上使用 nssm（Non-Sucking Service Manager）将 python main_ws.py 注册为系统服务，崩溃自动重启。Linux 上用 systemd Restart=always。无需改代码。
健康检查端点 低（1 小时） 在 main_ws.py 中加一个 HTTP server（http.server 标准库），监听本地端口，返回 {"status":"ok","uptime":...}。配合 nssm/systemd 做健康检查，挂了自动拉起来。
日志持久化 + 轮转 低（30 分钟） 将 stdout 日志改为 logging.handlers.RotatingFileHandler，保留最近 7 天、单文件 50MB。不在终端里 grep 历史日志。
关键指标埋点 中（2 小时） 不引入 Prometheus。在 ToolExecutor.execute() 入口/出口打点，输出到独立 metrics.log：{ts, tool, duration_ms, status}。每天跑个 Python 脚本统计各工具调用量、失败率、P95 延迟。够用。

P1 · 工程化成熟度

改进项投入方案
Prompt 评测集 中（1 天） 从历史日志中提取 50 条真实消息 + 预期行为（该调哪个工具 / 该走哪个路由分类 / 该拒绝还是执行）。每次改 System Prompt 后跑一遍，diff 行为差异。
格式：{"input":"帮我约张三","expected_route":"write","expected_tools":["search_users"],"forbidden_tools":[]}
不需要训练模型，正则 match 工具名 + 路由分类即可打分。
A/B 灰度框架 中（半天） 在 assistant.py 的 process() 入口加一个按 open_id 取模的分流逻辑：hash(open_id) % 100 < 20 → 走新版 Prompt，其余走旧版。对比两组的关键指标（工具调用成功率、用户"重试"频率）。20% 灰度 + 对比一周后全量。
Docker 化 低（1 小时） 写一个 Dockerfile（FROM python:3.9-slim + COPY + pip install + CMD）。Windows 上跑 Linux 容器需要 WSL2，或者直接在 Linux 服务器上 build。目的是让环境可复现、部署可一键。
代码 Hook 的单元测试 中（半天） 优先覆盖：_classify_request()（路由分类，输入输出明确，最值得测）、tool_input → JSON Schema 校验（参数缺失/类型错误场景）、Token 刷新逻辑（过期边界）。用 pytest + unittest.mock 替换 requests 调用。不追求覆盖率，覆盖核心逻辑即可。
操作审计日志 低（1 小时） 在 ToolExecutor.execute() 中追加一条结构化日志到 audit.log：{ts, user_open_id, tool, params_summary, result_summary}。不存完整参数（避免敏感数据），只存摘要。用于事后追溯"谁在什么时间做了什么操作"。

P2 · 规模化支撑

改进项投入方案
容量评估 + 瓶颈突破 中（1 天） 50 人的瓶颈在 Claude API 并发（Anthropic 默认 RPM 限制），不在 WebSocket。扩展到 500 人需要：① 消息队列（内存 queue.Queue 缓冲峰值）；② API 并发控制（asyncio.Semaphore 限流）；③ SQLite → PostgreSQL（写并发超过 SQLite 单写锁极限时需要）。先做压测确认瓶颈再动手。
VIN 数据脱敏 低（30 分钟） 在日志和 SQLite 写入前对 VIN 做截断（只保留前 11 位，后 6 位替换为 ******），故障码和诊断结论不脱敏（不含 PII）。17 位 VIN 前 11 位是 WMI+VDS（厂牌+车型），后 6 位是序列号（可关联到车主），只需遮后 6 位。
成本归因仪表 低（1 小时） 在每个 Tool Use 轮次记录 token 消耗（从 Claude API response 的 usage 字段提取），按工具名/skill 归类。不建仪表盘，导出 CSV 用 Excel 做透视表——知道"故障诊断"和"会议邀约"各花了多少钱，才能判断哪个 Skill 值得继续优化 Prompt 压缩 token。
多 Bot 实例 + 消息路由 高（3 天） 当单实例瓶颈到达时（WebSocket 单连接上限 ~5000 事件/秒，当前远未触及），拆分 Bot：一个处理办公场景（会议/文档/群管理），一个处理诊断场景（故障码/Bag/国标），通过飞书事件订阅的规则过滤分流到不同 WebSocket URL。各自独立部署，故障隔离。

设计哲学

📝

Prompt 是代码

SKILL.md 与 run_tool.py 同等重要。Prompt 定义行为边界，代码实现执行逻辑。两者分离、各自迭代。

🔧

工具是接口

Tool Schema 的 description 即 API 文档——Claude 不看代码只看描述。写 description 花的时间比写实现多。

🔄

规则是迭代出来的

19 条 System Prompt 格式规则没有一条是凭空设计的——每条对应一个生产环境中 Claude 的"坏习惯"。

一句话总结：这个 Agent 系统的价值不在于"用了 Claude API + 飞书 SDK"，而在于 10 个月持续迭代中积累的工程判断——什么时候该用规则而不是模型、什么时候该加锁而不是改代码、 什么时候该在 Prompt 里加一条硬约束而不是期望 LLM 自己理解。这些细节决定了 Agent 是「Demo」还是「生产可用」。
